What The Current Build Actually Does

The product is no longer just “sign recognition in the browser”. It is a split system with clear roles and fallbacks.

🗣️

Speaker-Side Subtitles

The speaker uses browser speech recognition. Their words are shown as live subtitles in Meet and sent as finalized transcript chunks for downstream ASL planning.

🔄

Room Relay Sync

The Node relay server groups participants by room, relays captions between peers, and broadcasts sign-plan payloads so the right participant sees the right output.

🤟

ASL Planner With Fallback

The FastAPI sign service tries known ASL phrases first, then approved sign units, then fingerspelling. That means “Hello” becomes a whole sign while names like “Maya” become M-A-Y-A.

Runtime Architecture

Three pieces work together during a call.

1. Meet Extension

Role-based UI in the browser

The extension injects the Meet overlay, lets users pick Speaker or ASL Signer, shows transcript history, and mounts the ASL player only on the receiving ASL side.

  • Speaker: microphone speech to live transcript
  • Signer: transcript feed plus ASL playback panel
  • Single-device preview supported for local testing
2. Relay Server

Room-based WebSocket transport

The `server/` service handles `JOIN`, `CAPTION`, `SIGN_PLAN`, `PEER_JOINED`, and `PEER_LEFT` events. It keeps each Meet room isolated and forwards live updates to the other participant.

  • HTTP health endpoint
  • Rate limiting and room size guardrails
  • Peer-to-peer sync without storing conversations
3. Sign Service

Phrase planning over raw translation

The `sign-service/` API accepts finalized text at `POST /api/sign-plan` and returns a sign playback manifest. It exposes provider metadata, checks known ASL phrase mappings, and falls back to fingerspelling for unknown terms.

  • `HELLO` and other known phrases resolve as full signs
  • Single unknown words can fingerspell letter by letter
  • Optional `sign-language-translator` integration is supported

Call Flow

1

Speaker talks

Speech recognition produces interim and final captions in Meet.

2

Relay forwards

The room server sends transcript updates and sign-plan events to the other participant.

3

Planner resolves

The sign service returns a whole sign when it knows the phrase and fingerspelling when it does not.

4

Signer receives

The ASL-side participant sees the ASL playback panel while the speaker keeps on-camera subtitles.

Run The Stack

The landing page now matches the actual local setup required by the current MVP.

1. Start the relay

cd server
npm start

Default URL: ws://172.20.10.8:3001

2. Start the sign service

cd sign-service
source .venv/bin/activate
python -m uvicorn app:app --host 172.20.10.8 --port 8001

API endpoint: POST /api/sign-plan

3. Load the extension

chrome://extensions
Load unpacked
Select the repo root

Set relay and planner URLs in the popup, then join the same Meet room on both sides.
